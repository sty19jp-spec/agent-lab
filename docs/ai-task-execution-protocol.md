# AI Task Execution Protocol (Layer5 Phase25)

## 1. Purpose
Define the runtime contract for AI task execution in `agent-lab`.

The protocol formalizes how a task is accepted, validated, executed, and recorded as repository-visible evidence.
It is designed to be lightweight, deterministic, and compatible with the existing runtime pipeline.

## 2. Scope
This protocol applies to runtime execution from `runtime.entry` through `runtime.engine` and `runtime.evidence`.

Included:
- execution pipeline contract
- task input contract
- preflight contract
- task output and artifact contract
- execution evidence contract
- operator contract

Not included:
- architecture redesign of runtime modules
- external orchestration systems
- approval UI workflows

## 3. Runtime Execution Pipeline
Canonical pipeline:

```text
task package
-> bundle resolve
-> preflight
-> execution
-> execution evidence
```

Current module mapping:
- `runtime.entry`: CLI arguments and runtime invocation
- `runtime.loader`: load/validate loader contract and task/bundle schema keys
- `runtime.discovery`: deterministic resolution of task/bundle refs
- `runtime.policy`: Gate A and Gate B preflight policy evaluation
- `runtime.engine`: task execution orchestration and status transitions
- `runtime.evidence`: evidence object construction and persistence

## 4. Runtime Execution Contract
### 4.1 Entrypoint Contract
Required CLI parameters:
- `--task-package-ref`
- `--runtime-bundle-ref`
- `--trigger-type`
- `--requested-operator`

Optional parameter:
- `--retry-counter` (default `0`)

Return contract:
- process exits `0` when execution success is `true`
- process exits `1` when blocked or failed
- runtime prints final evidence JSON to stdout

### 4.2 Loader Contract
Canonical loader object (`runtime.loader.LoaderContract`):

```json
{
  "task_package_ref": "task://docs-validation",
  "runtime_bundle_ref": "bundle://local-docs-validator",
  "trigger_type": "manual",
  "requested_operator": "Executor"
}
```

Validation rules:
- all four fields are required and must be non-empty strings
- task yaml and bundle yaml must contain required top-level keys

Required `task.yaml` keys:
- `task_id`
- `task_type`
- `operator`
- `bundle`
- `contract`
- `input`
- `output`

Required `bundle.yaml` keys:
- `bundle_id`
- `bundle_version`
- `executor`
- `resources`
- `policy`

## 5. Task Input Contract
Task input is declared in `task.yaml` and resolved against repository context.

Minimal input contract shape:

```yaml
input:
  target_files:
    - README.md
    - docs/AI-task-protocol.md
```

Current task-type-specific rule (`docs_validation`):
- `input.target_files` must be a non-empty list
- each entry is treated as a path string and resolved by runtime

Path resolution behavior (`runtime.engine._to_repo_path`):
1. absolute path: use as-is
2. relative path under task directory: use if exists
3. otherwise resolve against repository current working directory

## 6. Preflight Contract
Preflight is mandatory before execution.

### 6.1 Gate A (Canonical Set)
Pass conditions:
- `trigger_type` is one of `manual`, `schedule`, `event_stub`
- `requested_operator` is one of `Architect`, `Executor`, `Auditor`, `Human`

### 6.2 Gate B (Runtime Policy)
Pass conditions:
- task package is resolvable
- runtime bundle is resolvable
- task declares a bundle ref
- declared task bundle matches requested runtime bundle (ref or resolved file)
- `task.operator == requested_operator`
- `bundle.executor == requested_operator`
- for `event_stub`, `requested_operator` must be `Executor`

Preflight result contract:
- both gate result objects are emitted with `gate`, `passed`, and `reason`
- execution proceeds only when Gate A and Gate B both pass

## 7. Task Output Contract
Task output is declared in `task.yaml` under `output`.

Example output contract:

```yaml
output:
  report_path: examples/evidence/docs-validation-report.json
  evidence_path: examples/evidence/docs-validation-evidence.json
```

Runtime behavior:
- task action may write task-specific report artifact(s)
- runtime always writes execution evidence JSON
- when `output.evidence_path` is present, runtime evidence is written to that path
- otherwise fallback evidence path is `examples/evidence/<run_id>.json`

## 8. Artifact Contract
Artifacts are repository-visible files generated by task execution.

Current artifact categories:
- task artifact: task-specific output (for example validation report)
- runtime evidence artifact: execution evidence JSON

### 8.1 Task Artifact Structure (docs_validation)
Current `docs_validation` report shape:

```json
{
  "task_id": "docs_validation_minimal",
  "task_type": "docs_validation",
  "bundle_id": "local-docs-validator",
  "bundle_version": "0.1.0",
  "checked_at": "2026-01-01T00:00:00+00:00",
  "checks": [
    {
      "path": "/abs/path/to/file.md",
      "exists": true,
      "is_markdown": true,
      "size_bytes": 123
    }
  ],
  "missing_count": 0
}
```

Artifact requirements:
- JSON format
- deterministic keys for same runtime version/task type
- persisted in repository workspace for git/PR inspection

## 9. Execution Evidence Contract
Runtime evidence is the canonical audit object produced by each run.

Top-level structure (`runtime.evidence.build_evidence`):

```json
{
  "task_evidence": {},
  "execution_evidence": {},
  "execution_summary": {},
  "preflight_summary": {},
  "close_summary": {},
  "run_state": {},
  "evidence_file": "/abs/path/to/evidence.json"
}
```

### 9.1 `task_evidence`
Required fields (current runtime):
- `task_ref`
- `task_resolved`
- `task_id`
- `task_type`
- `bundle_ref`
- `bundle_resolved`
- `bundle_id`
- `bundle_version`

### 9.2 `execution_evidence`
Required fields:
- `success` (bool)
- `action` (string)
- `operator` (string)
- `detail` (string)
- `outputs` (object)

### 9.3 `preflight_summary`
Required fields:
- `passed` (bool)
- `gate_a` (`gate`, `passed`, `reason`)
- `gate_b` (`gate`, `passed`, `reason`)

### 9.4 `close_summary`
Required fields:
- `run_id`
- `status`
- `dedup_key`
- `retry_counter`

### 9.5 `run_state`
Required fields:
- `run_id`
- `status`
- `dedup_key`
- `retry_counter`
- `started_at`
- `ended_at`
- `error`

Evidence guarantees:
- one evidence JSON per run outcome
- includes blocked, failed, and successful outcomes
- persisted to repository-visible path

## 10. Operator Contract
Operator names are part of canonical runtime policy and must be explicit.

Canonical operators:
- `Architect`
- `Executor`
- `Auditor`
- `Human`

Operator enforcement rules:
- requested operator must be canonical (Gate A)
- task operator binding must match requested operator (Gate B)
- bundle executor binding must match requested operator (Gate B)
- `event_stub` trigger is currently restricted to `Executor`

Execution mode alignment:
- default Executor mode is Codex CLI autonomous execution equivalent to `--ask-for-approval never`
- no interactive approval UI in normal runtime flow
- escalate only for destructive or security-related operations

## 11. Governance Alignment
This protocol remains repository-first and consistent with existing controls:
- GitHub `main` is source of truth
- changes are reviewed through branch + PR workflow
- artifacts and evidence remain inspectable in git history/diff
- preflight gates prevent off-contract execution from proceeding

Operational safety boundary:
- destructive/security-sensitive operations require human escalation
- other scoped runtime operations proceed autonomously under documented policy

## 12. Compatibility With Existing Runtime
This protocol is compatible with current modules and data shapes:
- no runtime module replacement is required
- no approval UI dependency is introduced
- no external control plane is required

Phase25 intent:
- specification-first formalization of the existing runtime contract
- keep implementation lightweight and practical
